# BeAR and Exo Skryer Opacity/Correlated-k Review

Date: 2026-07-12

## Scope

This review inspected the upstream source trees rather than relying only on
method descriptions:

- BeAR commit `0dc026fc8117ddbead2cab1953c155f82feca133`
- Exo Skryer commit `abde13e1e39753f0528e271227decec503211c64`

Both projects are copyleft licensed (GPL-3.0 and AGPL-3.0 respectively), so
ROBERT must not copy their implementation into its MIT-licensed source. The
algorithmic observations below can inform an independent implementation backed
by published methods and ROBERT's own parity tests.

## BeAR

BeAR's main opacity path is not correlated-k and does not perform random
overlap. It uses tabulated absorption cross sections, normally generated by
HELIOS-k on a common 0.01 cm-1 source grid. At setup it selects only the
wavenumbers required by the configured observation range and chosen constant
wavenumber, wavelength, or resolving-power grid.

Its speed comes from systems and data-layout choices:

- a C++ core compiled with `-O3` and OpenMP;
- optional NVIDIA CUDA, which the upstream documentation strongly recommends;
- single-precision opacity files and float GPU kernels;
- persistent sampled cross sections on the device;
- batched pressure/temperature interpolation across species and layers;
- one GPU thread per spectral point, with species contributions accumulated
  directly into the absorption coefficient;
- GPU RT, convolution, band integration, and likelihood kernels, avoiding
  repeated host/device transfers;
- contribution functions computed as an optional post-processing product.

This architecture explains its throughput, but it does not offer a faster
random-overlap algorithm transferable to ROBERT. It is closer to ROBERT's beta
opacity-sampling path, with substantially more aggressive accelerator and
single-precision engineering.

## Exo Skryer

Exo Skryer supports opacity sampling and correlated-k. Its correlated-k path
offers three gas-combination approaches:

1. RORR: sequential random overlap, sorting all `n_g**2` pairs after adding
   each species.
2. PRAS: a deterministic-seeded Latin-hypercube approximation that samples
   the species distributions before sorting.
3. TRANS: for transmission spectra only, multiply each species' mean
   transmission and avoid constructing a mixed k-distribution.

The standard RORR implementation:

- precomputes the outer-product g weights;
- uses a sequential `lax.scan` over species;
- uses `lax.sort_key_val` for the complete pair distribution;
- resamples by interpolating `log10(k)` at the configured g-points;
- vectorizes independent layers and wavelengths with `jax.vmap`;
- interpolates and mixes one layer at a time to avoid materializing the full
  species/layer/wavelength/g array.

Most of Exo Skryer's acceleration is broader than this kernel. It JIT-compiles
the complete atmosphere, opacity, RT, instrument, and likelihood closure with
JAX/XLA and keeps static opacity caches in JAX arrays. It can run that compiled
graph on a multi-threaded CPU or GPU.

Exo Skryer's resampling is not numerically identical to ROBERT's conservative
target-bin integration. PRAS and transmission multiplication are explicitly
different algorithms and must be treated as new scientific methods, not
drop-in performance backends.

## Controlled CPU comparison

A warmed comparison used actual WASP-69b F322W2 g ordinates and six molecular
k-tables, expanded to 80 layers and 144 wavelength bins. Both calculations ran
with eight CPU threads and excluded JIT compilation.

| Kernel | Median time |
|---|---:|
| ROBERT Numba heap RORR | 29 ms |
| Exo Skryer JAX/XLA RORR | 494 ms |

The Exo Skryer mixed k-distribution differed from ROBERT's conservative result
by 3.79% RMS and up to 32.6% at individual k ordinates in this controlled
calculation. This is a kernel comparison, not a full spectral validation, and
the difference is expected from the distinct resampling definitions.

The result rules out replacing ROBERT's current CPU heap kernel with Exo
Skryer's full-sort/log-interpolation RORR implementation. Exo Skryer may still be
faster for a complete retrieval on a GPU because its principal advantage is
whole-model compilation and device residency.

## Recommended ROBERT experiments

### 1. Compiled flat-array likelihood core

Keep Python domain objects at setup boundaries, but pack the changing
atmosphere and cached opacity into contiguous arrays. Compile one call spanning
opacity interpolation, gas mixing, absorption-only RT, instrument binning, and
the Gaussian likelihood. This transfers Exo Skryer's most important idea while
preserving ROBERT's public API and reference implementations.

### 2. Transmission-product emission experiment

For constant source function within a layer, random-overlap expectations of
transmission factor exactly by species. That suggests an absorption-only
emission experiment that multiplies per-species transmissions rather than
resorting and rebinning k-distributions. ROBERT's current exact linear-in-
optical-depth source treatment prevents assuming equivalence without proof.
The experiment must therefore compare:

- brute-force Cartesian g combinations on small fixtures;
- the current RORR reference;
- the constant-source transmission-product formulation;
- the full WASP-69b spectrum and likelihood over representative prior points.

It should remain experimental unless its error is comfortably below both the
current correlated-k representation error and observational uncertainty.

### 3. Precision study, not an immediate float32 switch

BeAR obtains substantial accelerator throughput from float32. ROBERT should
benchmark a mixed-precision kernel separately, including weak-opacity tails,
transmission, eclipse depth, likelihood, and posterior-level effects. The
float64 scientific reference must remain unchanged.

### 4. Retrieval-level batching

Whole-model compilation is most valuable when several independent likelihood
points can be evaluated together. Before adopting JAX as another backend,
benchmark parallel UltraNest likelihood calls and batched parameter vectors
against the current Numba kernel to determine whether the workload is large
enough to amortize compilation and dispatch costs.

## Decision

Do not replace ROBERT's RORR kernel with either inspected implementation.
BeAR's speed is an opacity-sampling/CUDA result, while Exo Skryer's RORR is
slower on the tested CPU and uses a different resampling definition. The best
next prototype is the transmission-product absorption-emission formulation,
run as a scientific experiment alongside a flat-array compiled likelihood
benchmark.
